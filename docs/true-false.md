# لعبة صح أو خطأ (True or False) — توثيق كامل

هذا الملف يوثّق لعبة "صح أو خطأ" بالتفصيل: بنية البيانات، كل الـ endpoints المتاحة، أمثلة الطلبات والاستجابات، نظام الـ `id`، آلية منع التكرار (`random-exclude`)، وحالات الأخطاء.

هذه اللعبة مبنية بنفس أسلوب لعبة الكويز (`quiz`) الموجودة في المشروع بالضبط — نفس بنية الملفات، نفس شكل الاستجابات، ونفس نظام الأخطاء. إذا كنت مطّلعًا على توثيق الكويز في [`README.md`](../README.md)، هذا الملف سيبدو مألوفًا جدًا.

## وصف اللعبة

"صح أو خطأ" لعبة بسيطة: يُعرض على اللاعب عبارة (`question`)، وعليه تحديد ما إذا كانت صحيحة أم خاطئة. كل سؤال له إجابة صحيحة واحدة فقط ممثّلة بقيمة منطقية (`answer: true` أو `answer: false`)، وتصنيف موضوعي (`category`) مثل "علوم" أو "جغرافيا" أو "تاريخ".

الـ API هنا **مزوّد محتوى فقط** — لا يتحقق من إجابة المستخدم ولا يحتفظ بأي حالة (state) عن تقدّم اللعبة؛ هذا كله مسؤولية البوت المستهلك للـ API. الـ API بلا حالة (stateless) تمامًا، بنفس فلسفة لعبة الكويز.

## بنية البيانات

مصدر البيانات الأساسي هو ملف:

```
src/data/true-false/questions.json
```

عبارة عن مصفوفة JSON تحتوي حاليًا **420 سؤال** حقيقي بالعربية، موزّعة على عدة تصنيفات (علوم، تاريخ، فضاء، جغرافيا، حيوانات، نباتات، تقنية، رياضة، أنمي، ألعاب، ثقافة عامة، كيمياء، أفلام ومسلسلات، جيولوجيا، محيطات، اختراعات، أدب، فنون).

شكل كل سؤال كما يظهر في استجابات الـ API:

```json
{
  "id": 1,
  "question": "الأرض هي الكوكب الثالث من حيث البعد عن الشمس.",
  "answer": true,
  "category": "فضاء"
}
```

| الحقل | النوع | الوصف |
|---|---|---|
| `id` | `number` | معرّف فريد وثابت، **مخزَّن يدويًا داخل questions.json** (ليس مولَّدًا تلقائيًا). انظر [نظام الـ id](#نظام-الـ-id) بالأسفل. |
| `question` | `string` | نص العبارة المطلوب تحديد صحتها. يدعم العربية وUTF-8 بالكامل. |
| `answer` | `boolean` | الإجابة الصحيحة: `true` (العبارة صحيحة) أو `false` (العبارة خاطئة). قيمة منطقية واحدة فقط، وليست مصفوفة. |
| `category` | `string` | تصنيف موضوعي حر (مثل `"علوم"`, `"جغرافيا"`). |

> **ملاحظة:** على عكس لعبة الكويز في هذا المشروع، مصدر بيانات "صح أو خطأ" **يحتوي حقل `id` صراحة** داخل كل سؤال في `questions.json` نفسه، وليس مولَّدًا في طبقة الخدمة أثناء التحميل.

أي سجل ناقص أو غير صالح (مثلًا `question` فارغ، أو `answer` ليس قيمة منطقية فعلية، أو `category` مفقود) يُتجاهل تلقائيًا وقت التحميل ولن يظهر في أي endpoint — دون أن يوقف الـ API بالكامل.

## نظام الـ id

- الـ `id` **رقم صحيح موجب** (وليس نصًا من نوع `"tf_123"`)، بنفس أسلوب `id` في لعبة الكويز من ناحية الشكل — للحفاظ على اتساق كامل مع بقية الـ API.
- على عكس لعبة الكويز، الـ `id` هنا **مخزَّن يدويًا/مسبقًا داخل كل سؤال في `questions.json` نفسه** — وليس مولَّدًا تلقائيًا وقت التحميل حسب ترتيب السؤال في الملف. كل سؤال في الملف يحتوي حقل `id` صريح:
  ```json
  { "id": 1, "question": "...", "answer": true, "category": "..." }
  ```
- **السبب في هذا الاختيار:** لو كان الـ `id` يتولّد تلقائيًا حسب الترتيب (زي الكويز)، فإن أي عملية استيراد لأسئلة جديدة في **منتصف** الملف كانت ستغيّر `id` كل الأسئلة التي تليها — وهذا يكسر أي قائمة استثناء (`random-exclude`) مخزّنة مسبقًا لدى البوت، ويكسر أي رابط `/api/games/true-false/:id` كان محفوظًا سابقًا. تخزين `id` يدويًا وثابتًا داخل الملف نفسه يحل هذه المشكلة جذريًا: أي سؤال يحتفظ بنفس `id`ه إلى الأبد، بغض النظر عن ترتيبه أو عدد الأسئلة المضافة حوله.
- الـ `id` **ثابت بشكل دائم** لكل سؤال طالما لم يُعدَّل يدويًا. هذا يجعله موثوقًا للاستخدام في:
  - تتبّع الأسئلة التي عُرضت على مستخدم/مجموعة معيّنة (لمنع التكرار عبر البوت).
  - جلب نفس السؤال مرة أخرى بواسطة `GET /api/games/true-false/:id`.
  - **منع تكرار الأسئلة وقت عملية الاستيراد نفسها** (الهدف الأساسي من هذا النظام): بما أن كل سؤال له `id` ثابت ومعروف مسبقًا، يمكن مقارنة الأسئلة الجديدة المطلوب استيرادها بالأسئلة الموجودة فعليًا (بمقارنة نص السؤال أو أي معرّف خارجي آخر) قبل توليد `id` جديد لها، بدل الاعتماد فقط على ترتيب الظهور في الملف.

### قواعد إضافة أسئلة جديدة (استيراد)

عند إضافة أسئلة جديدة (سواء يدويًا أو عبر عملية استيراد آلية):

1. **لازم كل سؤال جديد يحصل على `id` فريد لم يُستخدم من قبل في الملف بالكامل.** الطريقة الأبسط: خذي أكبر `id` موجود حاليًا في `questions.json` وابدئي الترقيم من بعده (مثلًا لو أكبر `id` حاليًا هو `420`، فالسؤال الجديد الأول يأخذ `id: 421`، والذي يليه `id: 422`، وهكذا).
2. **لا تُعيدي استخدام `id` سؤال تم حذفه سابقًا** إلا إذا كنتِ متأكدة تمامًا أنه لا يوجد أي بوت أو نظام خارجي لا يزال يشير لهذا الـ `id` القديم في قائمة استثناء محفوظة لديه.
3. طبقة الخدمة (`true-false.service.ts`) تتحقق تلقائيًا من فرادة الـ `id` وقت التحميل: لو وُجد `id` مكرر بين سؤالين، **يُحتفَظ فقط بأول سؤال بهذا الـ `id` ويُتجاهل الباقي بصمت** (مع تسجيل تحذير في الـ server logs)، بدل أن يوقف الـ API بالكامل أو يرجع نتائج غير متوقعة.
4. لتفادي أي التباس، يُفضَّل دائمًا إضافة الأسئلة الجديدة في **نهاية الملف** (رغم أن ترتيبها الفعلي في الملف لم يعد يؤثر على الـ `id` بعد الآن، بما أن الـ `id` مخزَّن صراحة وليس مشتقًا من الترتيب).

## الـ Endpoints

جميع الـ endpoints التالية تعمل عبر بنية المشروع العامة `/api/games/[game]/...` باستخدام `true-false` كاسم اللعبة (slug). هذه نفس البنية المستخدمة تمامًا للعبة الكويز — فقط استبدلي `quiz` بـ `true-false` في المسار.

### `GET /api/games/true-false/random`

ترجع سؤال "صح أو خطأ" عشوائيًا واحدًا.

**مثال الطلب:**

```bash
curl https://raygumo-api.vercel.app/api/games/true-false/random
```

**مثال الاستجابة (200):**

```json
{
  "success": true,
  "data": {
    "id": 123,
    "question": "الشمس أكبر من الأرض.",
    "answer": true,
    "category": "علوم"
  }
}
```

---

### `GET /api/games/true-false/random-exclude`

ترجع سؤالًا عشوائيًا واحدًا، مع استثناء أي `id` موجود ضمن قائمة معامل الاستعلام `ids` (مفصولة بفواصل). هذا هو آلية منع التكرار (anti-repeat) في اللعبة، وهي مطابقة تمامًا لفكرة `random-exclude` الموجودة في Quiz API.

> **ملاحظة على اسم المعامل:** بنفس اتساق مسار الكويز في هذا المشروع، اسم معامل الاستعلام هو `ids` (وليس `exclude`):
> `GET /api/games/true-false/random-exclude?ids=1,2,3`

**لماذا هذا الـ endpoint موجود:** الـ API بلا حالة تمامًا — لا يخزّن أي معلومة دائمة عن الأسئلة التي عُرضت سابقًا. البوت (مثلًا بوت واتساب) هو المسؤول عن تتبّع الأسئلة المستخدمة لكل مستخدم/مجموعة، ثم إرسال قائمة تلك الـ `id` مع كل طلب حتى لا يتكرر نفس السؤال.

**مثال الطلب:**

```bash
curl "https://raygumo-api.vercel.app/api/games/true-false/random-exclude?ids=1,2,3"
```

**مثال الاستجابة (200) — سؤال غير موجود ضمن `1,2,3`:**

```json
{
  "success": true,
  "data": {
    "id": 42,
    "question": "الحوت الأزرق يُعد من الثدييات.",
    "answer": true,
    "category": "حيوانات"
  }
}
```

**سلوكيات خاصة:**

- إذا كان معامل `ids` غير موجود أو فارغًا، يعمل هذا الـ endpoint تمامًا مثل `/random` العادي (استثناء لا شيء = كل الأسئلة متاحة).
- إذا استُثنيت **كل** الأسئلة المتاحة (لم يتبقَّ أي سؤال جديد)، يرجع الـ endpoint خطأ `404 NOT_FOUND` واضحًا بدل تكرار سؤال قديم بصمت. على البوت عندها تصفير قائمة الأسئلة المستخدمة والمحاولة من جديد.

**مثال استجابة عند استنفاد كل الأسئلة (404):**

```json
{
  "success": false,
  "message": "لا يوجد أي سؤال متبقٍ بعد استثناء كل الأسئلة المُمرَّرة (كل الأسئلة استُخدمت بالفعل)",
  "code": "NOT_FOUND"
}
```

**نمط مقترح للتكامل من البوت (anti-repeat):**

1. احتفظي بقائمة لكل مستخدم/مجموعة بأرقام الأسئلة المستخدمة (في تخزين البوت نفسه).
2. نادي على `GET /api/games/true-false/random-exclude?ids=<الأرقام المستخدمة مفصولة بفواصل>` بدل `/random` العادي.
3. أضيفي `id` السؤال المُرجَع لقائمة الأرقام المستخدمة.
4. إذا رجع الـ endpoint `404 NOT_FOUND` (كل الأسئلة استُنفدت)، صفّري القائمة ونادي مرة أخرى.

### `GET /api/games/true-false/count`

ترجع العدد الإجمالي لأسئلة "صح أو خطأ" المتاحة حاليًا.

**مثال الطلب:**

```bash
curl https://raygumo-api.vercel.app/api/games/true-false/count
```

**مثال الاستجابة (200):**

```json
{
  "success": true,
  "data": {
    "count": 420
  }
}
```

> هذا الشكل (`data.count` بدل `count` على المستوى الأعلى) يتّبع الشكل الموحّد لكل استجابات الـ API في هذا المشروع (`{ success, data }` دائمًا) — راجعي [شكل الاستجابة الموحّد](#شكل-الاستجابة-الموحد) بالأسفل.

### `GET /api/games/true-false/:id`

ترجع سؤالًا واحدًا محددًا بالضبط عبر رقمه (`id`)، بدل سؤال عشوائي. مفيد عند إعادة عرض نفس السؤال، أو بناء نظام تصفح تسلسلي.

**مثال الطلب:**

```bash
curl https://raygumo-api.vercel.app/api/games/true-false/123
```

**مثال الاستجابة (200):**

```json
{
  "success": true,
  "data": {
    "id": 123,
    "question": "الشمس أكبر من الأرض.",
    "answer": true,
    "category": "علوم"
  }
}
```

**مثال استجابة عند عدم وجود السؤال (404):**

```json
{
  "success": false,
  "message": "No item with id 999999 in \"true-false\"",
  "code": "NOT_FOUND"
}
```

**مثال استجابة عند `id` غير صالح (400):**

```json
{
  "success": false,
  "message": "Invalid id parameter: \"abc\"",
  "code": "BAD_REQUEST"
}
```

## شكل الاستجابة الموحّد

كل endpoint في هذه اللعبة يتّبع نفس الشكل الموحّد المستخدم في كل أنحاء المشروع (راجعي `src/types/api.ts`):

**نجاح:**

```json
{
  "success": true,
  "data": { }
}
```

**خطأ:**

```json
{
  "success": false,
  "message": "نص يشرح الخطأ",
  "code": "NOT_FOUND"
}
```

## حالات الأخطاء

| الحالة | الكود (`code`) | حالة HTTP | متى تحدث |
|---|---|---|---|
| السؤال غير موجود | `NOT_FOUND` | 404 | `:id` رقمي صحيح لكن لا يوجد سؤال بهذا الرقم |
| مجموعة الأسئلة فارغة | `NOT_FOUND` | 404 | `questions.json` فارغ أو كل سجلاته غير صالحة |
| كل الأسئلة مستثناة | `NOT_FOUND` | 404 | `random-exclude` استُثنيت فيه كل الأسئلة المتاحة عبر `ids` |
| `id` غير صالح | `BAD_REQUEST` | 400 | قيمة `:id` ليست رقمًا صحيحًا موجبًا (مثل `"abc"` أو `"-1"`) |
| قيمة غير صالحة في `ids` | `BAD_REQUEST` | 400 | أحد عناصر معامل `ids` في `random-exclude` ليس رقمًا صحيحًا موجبًا |
| اسم لعبة غير معروف | `NOT_FOUND` | 404 | حدث فقط لو استُخدم اسم مختلف عن `true-false` في مسار غير متوقع |
| خطأ داخلي غير متوقع | `INTERNAL_ERROR` | 500 | مشكلة غير متوقعة في القراءة أو المعالجة |

مثال عام لكيفية التعامل مع أي رد من طرف البوت:

```js
async function callApi(path) {
  const res = await fetch(`https://raygumo-api.vercel.app${path}`);
  const json = await res.json();
  if (!json.success) {
    throw new Error(`[${json.code ?? "ERROR"}] ${json.message}`);
  }
  return json.data;
}
```

## إضافة/تحديث الأسئلة

لتحديث محتوى لعبة "صح أو خطأ":

1. عدّلي `src/data/true-false/questions.json` مباشرة (أضيفي/عدّلي/احذفي عناصر في المصفوفة).
2. التزمي بنفس شكل السجل: `id` (رقم صحيح موجب فريد)، `question` (نص غير فارغ)، `answer` (قيمة منطقية `true`/`false` فعلية)، `category` (نص غير فارغ).
3. **أعطي كل سؤال جديد `id` فريدًا لم يُستخدم من قبل** — راجعي [قواعد إضافة أسئلة جديدة (استيراد)](#قواعد-إضافة-أسئلة-جديدة-استيراد) أعلاه للتفاصيل الكاملة.
4. اعملي `commit` وارفعي (`redeploy`) — بدون أي تعديل كود مطلوب.

للحفاظ على وضوح الملف، يُفضَّل إضافة الأسئلة الجديدة في **نهاية الملف** مع `id` أكبر من أعلى `id` موجود حاليًا.
