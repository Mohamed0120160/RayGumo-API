# شرح معمارية RayGumo API — للمبتدئين تمامًا

هذا الملف يشرح المشروع بأبسط طريقة ممكنة، وكأنك أول مرة تسمع عن بناء
API. لا تحتاج أي خبرة سابقة لفهمه.

هذه النسخة من المشروع هي **MVP (منتج بحد أدنى قابل للاستخدام)**: خفيفة
جدًا، تخدم لعبة **كويز واحدة فقط** حاليًا، لكنها مبنية بحيث يسهل توسيعها
لاحقًا.

## ما هو الـ API أصلًا؟

**API** تعني "واجهة برمجة تطبيقات" (Application Programming Interface).
بكل بساطة: API هو **طريقة يتحدث بها برنامجان مع بعضهما البعض** عبر
الإنترنت.

مثلًا: بوت واتساب هو برنامج، وRayGumo API هو برنامج آخر. البوت "يسأل"
RayGumo API سؤالًا (مثل: "أعطني سؤال كويز عشوائي")، وRayGumo API "يرد"
عليه بالإجابة.

هذا التواصل يحدث عبر بروتوكول اسمه **HTTP** - نفس البروتوكول الذي
يستخدمه متصفحك لفتح المواقع. البوت يرسل "طلب" (request) لرابط معين،
والخادم يرد بـ"استجابة" (response).

## تشبيه بسيط: المطعم

تخيّل مطعمًا يقدّم طبقًا واحدًا فقط حاليًا (كويز)، لكن قائمته مصممة بحيث
يسهل إضافة أطباق جديدة لاحقًا:

- **الزبون** = بوت الواتساب (أو أي برنامج يستخدم الـ API)
- **قائمة الطعام (المنيو)** = مستندات الـ API (README) التي تخبرك أي
  "أطباق" (endpoints) متاحة وكيف تطلبها
- **النادل** = الـ API نفسه (RayGumo API) - يستقبل طلبك وينقله للمطبخ
- **المطبخ** = طبقة التخزين (ملف questions.json) - المكان الذي توجد
  فيه المكوّنات الفعلية
- **الطبق النهائي** = الاستجابة (response) التي تصل للزبون

عندما تطلب "سؤال كويز"، النادل (API) لا يطبخ بنفسه بشكل عشوائي - هو
يذهب للمطبخ (questions.json)، يحضّر الطلب بالطريقة الصحيحة، ثم يرجع لك
الطبق (JSON response) بشكل موحّد وأنيق في كل مرة.

## كيف يترجم هذا التشبيه لمشروعنا؟

```
┌─────────────────┐         ┌──────────────────┐        ┌────────────────┐
│  بوت الواتساب     │  طلب →  │   RayGumo API     │  قراءة │  questions.json │
│  (الزبون)         │ ←استجابة│   (النادل)         │ ← ────  │  (المطبخ)        │
└─────────────────┘         └──────────────────┘        └────────────────┘
```

الفكرة الأساسية: بدل أن يخزّن **كل** بوت واتساب أسئلة الكويز الخاصة به
بشكل منفصل، كل البوتات "تطلب" من نفس "المطعم" (RayGumo API)، وتحصل على
نفس المحتوى دائمًا وبنفس الشكل.

## الطبقات الأربع للمشروع

المشروع مبني من 4 طبقات، كل طبقة لها مسؤولية واحدة فقط ولا تتدخل في
مسؤوليات الطبقات الأخرى. هذا مبدأ برمجي مهم اسمه "فصل الاهتمامات"
(Separation of Concerns).

```
┌─────────────────────────────────────────────────────────┐
│  1. طبقة المسارات (Routes) — src/app/api/games/[game]/    │
│     "أستقبل طلب HTTP وأرجع استجابة HTTP"                   │
└───────────────────────┬─────────────────────────────────┘
                         │ يستدعي
                         ▼
┌─────────────────────────────────────────────────────────┐
│  2. طبقة السجلّ والوحدات — src/modules/games/              │
│     registry.ts: "أي لعبة هذه؟ أي وحدة أستخدم؟"             │
│     quiz/: "منطق الكويز الفعلي (اختيار، بحث، عدّ...)"         │
└───────────────────────┬─────────────────────────────────┘
                         │ يستدعي
                         ▼
┌─────────────────────────────────────────────────────────┐
│  3. طبقة التخزين (Storage) — src/lib/json-db.ts           │
│     "أقرأ ملفات JSON بأمان"                                │
└───────────────────────┬─────────────────────────────────┘
                         │ يقرأ من
                         ▼
┌─────────────────────────────────────────────────────────┐
│  4. طبقة البيانات (Data) — src/data/quiz/questions.json   │
│     "أنا المحتوى الفعلي: أسئلة الكويز"                      │
└─────────────────────────────────────────────────────────┘
```

**لماذا هذا الفصل مهم؟** لأنه يسمح لك بتغيير طبقة واحدة بدون كسر
الباقي. مثال: لو أردت لاحقًا إضافة لعبة "ألغاز"، تحتاج فقط تضيف وحدة
جديدة في الطبقة 2 (`src/modules/games/riddles/`) وملف بيانات جديد في
الطبقة 4 - الطبقتان 1 و3 لن تتغيرا إطلاقًا.

## رحلة طلب واحد، خطوة بخطوة

لنتتبّع ماذا يحدث بالضبط عندما يطلب بوت واتساب سؤال كويز عشوائي.

### الخطوة 1: المستخدم يكتب أمرًا

مستخدم على واتساب يكتب: `!كويز`

### الخطوة 2: البوت يقرر ماذا يفعل

بوت الواتساب (المبني بمكتبة Baileys) يرى الأمر `!كويز`، ويعرف من كوده
الخاص أنه يحتاج يطلب سؤالًا من RayGumo API.

### الخطوة 3: البوت يرسل طلب HTTP

```
GET https://raygumo-api.vercel.app/api/games/quiz/random
```

`GET` تعني "أعطني بيانات".

### الخطوة 4: الطلب يصل لـ RayGumo API

الملف المسؤول: `src/app/api/games/[game]/random/route.ts`

Next.js يفهم من الرابط `/api/games/quiz/random` أن `[game] = "quiz"`.

### الخطوة 5: الـ route يستدعي السجلّ (registry)

الـ route لا "يعرف" كيف يقرأ ملفات JSON بنفسه، ولا حتى يعرف تفاصيل
منطق الكويز - هو فقط يستدعي:

```ts
getRandomItem("quiz")
```

من ملف `src/modules/games/registry.ts`.

### الخطوة 6: السجلّ يتحقق ويفوّض المهمة لوحدة الكويز

`registry.ts` يتحقق: "هل quiz لعبة مسجّلة فعلًا؟" (نعم). ثم يستدعي
دالة وحدة الكويز:

```ts
quiz.getRandomQuestion()
```

### الخطوة 7: وحدة الكويز تحمّل البيانات وتولّد الـ id تلقائيًا

`quiz.service.ts` يستدعي `readRawCollection("quiz")` من
`src/lib/json-db.ts` لقراءة الملف الخام، ثم يولّد `id` تسلسليًا لكل
سؤال حسب ترتيبه في الملف، ثم يتحقق من صحة كل سؤال (عبر
`quiz.validation.ts`) قبل اختيار سؤال عشوائي منها.

### الخطوة 8: طبقة التخزين تقرأ الملف الفعلي

`json-db.ts` يفتح ويقرأ `src/data/quiz/questions.json`، يحوّله من نص
إلى قائمة (array) في الذاكرة، يختار سؤالًا عشوائيًا منها.

### الخطوة 9: البيانات تعود للأعلى، طبقة فطبقة

السؤال العشوائي يعود من `json-db.ts` إلى `quiz.service.ts` إلى
`registry.ts` إلى `route.ts`.

### الخطوة 10: الـ route يبني الاستجابة النهائية

باستخدام `lib/response.ts`:

```json
{
  "success": true,
  "data": {
    "id": 3,
    "question": "من كتب مسرحية روميو وجولييت؟",
    "answers": ["ويليام شكسبير"],
    "category": "أدب"
  }
}
```

### الخطوة 11: الاستجابة تعود للبوت

البوت يستقبل هذا الـ JSON، يقرأ `data.question`، ويرسله كرسالة واتساب:

```
❓ من كتب مسرحية روميو وجولييت؟
```

**كل هذه العملية تحدث في أجزاء من الثانية.**

## لماذا JSON وليس قاعدة بيانات؟

قد تتساءل: لماذا لا نستخدم قاعدة بيانات حقيقية مثل MySQL أو MongoDB من
البداية؟

### الإجابة المبسّطة

قاعدة البيانات تحتاج **خادمًا منفصلًا يعمل طوال الوقت**، وهذا يعني:
تكلفة إضافية، إعدادات أكثر تعقيدًا، وصيانة مستمرة. لمشروع MVP بحجم
RayGumo API - محتوى شبه ثابت (أسئلة كويز) لا يتغيّر كل ثانية - هذا
تعقيد غير ضروري.

**ملف JSON** هو ببساطة ملف نصي عادي، يُقرأ بسرعة، ولا يحتاج أي بنية
تحتية خاصة. يمكن تحديثه بسهولة تامة: فقط عدّل الملف واعمل `git push`.

### متى نحتاج قاعدة بيانات فعلية؟

عندما يحتاج المشروع بيانات تتغيّر كل ثانية، أو عمليات بحث معقدة جدًا،
أو كتابة متكررة جدًا من مستخدمين مختلفين في نفس اللحظة. لكن هذا **ليس**
حال RayGumo API - هو فقط "يقدّم" محتوى شبه ثابت للقراءة.

## ما معنى "بدون خادم" (Serverless)؟

### الطريقة التقليدية (خادم دائم)

تخيّل حاسوبًا (خادم) يبقى شغّالًا 24 ساعة، 7 أيام في الأسبوع، ينتظر
طلبات المستخدمين. حتى لو لم يأتِ أي طلب لمدة ساعة كاملة، الخادم يبقى
شغّالًا (ويكلّفك مالًا).

### الطريقة "بدون خادم" (Serverless) - وهي ما يستخدمه Vercel

بدل خادم واحد دائم التشغيل، كل "وظيفة" (function) - مثل الرد على طلب
`/api/games/quiz/random` - تعمل فقط **عندما يصل طلب فعلي**. بعد الرد،
"تنام" مرة أخرى.

**الفائدة:** توفير في التكلفة، وقابلية للتوسّع تلقائيًا.

## شكل الاستجابة الموحّد - ولماذا هو مهم

كل استجابة من RayGumo API، بدون استثناء، تأخذ أحد شكلين فقط:

### عند النجاح

```json
{
  "success": true,
  "data": { ... }
}
```

### عند الفشل

```json
{
  "success": false,
  "message": "رسالة توضح الخطأ",
  "code": "NOT_FOUND"
}
```

### لماذا هذا الاتساق مهم جدًا؟

تخيّل لو كل endpoint كان يرجع شكلًا مختلفًا - البوت الذي يتصل بالـ API
سيحتاج كودًا مختلفًا لكل endpoint. بفضل الشكل الموحّد، كود البوت يمكنه
دائمًا فعل نفس الشيء:

```js
const response = await fetch(apiUrl);
const result = await response.json();

if (result.success) {
  // استخدم result.data بثقة
} else {
  // اعرض result.message للمستخدم أو سجّله كخطأ
}
```

هذا النمط يعمل مع **أي** endpoint في المشروع بدون استثناء.

## لماذا "MVP"؟ ولماذا أُزيلت أجزاء من المشروع؟

### ما معنى MVP؟

MVP تعني "Minimum Viable Product" - "منتج بحد أدنى قابل للاستخدام".
الفكرة: تبني أبسط نسخة ممكنة من المشروع تعمل فعليًا وتحل المشكلة
الأساسية، بدل بناء كل الميزات الممكنة دفعة واحدة قبل التأكد أن الفكرة
الأساسية (خدمة أسئلة كويز عبر API) تعمل بشكل جيد.

### لماذا أُزيلت وحدات كانت موجودة سابقًا (AI، Tools، Admin)؟

في مرحلة سابقة من المشروع كانت هناك وحدات إضافية: ردود ذكاء اصطناعي
وهمية، أدوات فحص صحة الخادم، ولوحة أدمن محمية بمفتاح سري. تمت إزالتها
كلها لأنها **لم تكن مستخدمة فعليًا** - فقط بنية جاهزة بدون استخدام حقيقي
بعد. الاحتفاظ بكود غير مستخدم يزيد التعقيد بلا فائدة حقيقية، ويصعّب
على أي مطوّر جديد فهم المشروع بسرعة.

**القاعدة الذهبية للـ MVP:** ابدأ بأصغر نسخة تحل المشكلة الفعلية
(تقديم أسئلة كويز للبوتات)، ثم أضف الميزات الأخرى فقط عندما تصبح
مطلوبة فعليًا، وليس "لأنها قد تكون مفيدة يومًا ما".

### هل هذا يعني أن هذه الميزات لن تعود أبدًا؟

لا. يمكن إضافتها مجددًا مستقبلًا **عندما تصبح مطلوبة فعليًا**، وبنفس
الأسلوب المعماري النظيف (وحدة مستقلة، أنواع خاصة بها، خدمة خاصة بها).
لكن حاليًا، أبسط مشروع ممكن هو الأفضل.

## كيف تضيف شيئًا جديدًا؟ (خريطة سريعة)

| أريد إضافة... | اذهب إلى... |
|---|---|
| سؤال كويز جديد | `src/data/quiz/questions.json` (أضف عنصرًا في المصفوفة - بدون الحاجة لكتابة `id` يدويًا، فهو يُولَّد تلقائيًا حسب ترتيب السؤال في الملف) |
| استبدال كل الأسئلة بمجموعتي الخاصة | استبدل محتوى `src/data/quiz/questions.json` بالكامل |
| لعبة جديدة بالكامل (مثل ألغاز) | راجع قسم "إضافة لعبة جديدة مستقبلًا" في `PROJECT_GUIDE_AR.md` |

**القاعدة الذهبية**: لأي إضافة، اسأل نفسك "في أي طبقة من الطبقات
الأربع ينتمي هذا التغيير؟" - هذا سيوجهك مباشرة للملف الصحيح.

## أسئلة شائعة

### هل RayGumo API يخزّن معلومات عن المستخدمين (اللاعبين)؟

لا، إطلاقًا. المشروع مصمم عمدًا ليكون "مزوّد محتوى" فقط. أي معلومات عن
اللاعبين (نقاط، مستويات، تصنيفات) تبقى مسؤولية كل بوت على حدة.

### ماذا يحدث لو طلب البوت لعبة غير موجودة (مثل "anime")؟

سيحصل على استجابة خطأ 404 بالشكل الموحّد، لأن `anime` غير مسجّلة حاليًا
في `GAME_REGISTRY` (اللعبة الوحيدة المسجّلة الآن هي `quiz`):

```json
{
  "success": false,
  "message": "Unknown game category: \"anime\"",
  "code": "NOT_FOUND"
}
```

### هل يمكن لأكثر من بوت استخدام نفس الـ API في نفس الوقت؟

نعم بالضبط، وهذا هو الهدف الأساسي من المشروع. عدة بوتات يمكنها كلها
الاتصال بنفس الرابط والحصول على نفس المحتوى بنفس الشكل الموحّد، في
نفس اللحظة، بدون أي تعارض بينها.

### أين أعدّل أسئلة الكويز؟

في ملف واحد فقط: `src/data/quiz/questions.json`. عدّله، احفظ، اعمل
commit و push - لا حاجة لأي تعديل برمجي آخر. لا تحتاج لإضافة `id`
بنفسك عند إضافة سؤال جديد؛ يُولَّد تلقائيًا حسب ترتيب السؤال في الملف.

### كيف أمنع تكرار نفس السؤال على نفس المجموعة في الواتساب؟

الـ API نفسه **بدون حالة (stateless)** عمدًا - لا يتذكر أي أسئلة
استُخدمت سابقًا. بدل ذلك، استخدم مسار
`GET /api/games/quiz/random-exclude?ids=1,2,3` ومرّر قائمة الـ id التي
استخدمتها المجموعة سابقًا (يتتبّعها البوت بنفسه). لو استُنفدت كل
الأسئلة، يرجع المسار خطأ 404 واضحًا بدل تكرار سؤال قديم بصمت.

### هل يحتاج المشروع أي مفتاح سري أو كلمة مرور للعمل؟

لا. هذه نسخة MVP بدون أي مصادقة أو أسرار مطلوبة - كل الـ endpoints
عامة ومفتوحة للقراءة فقط (GET). لا يوجد `.env.local` إلزامي لتشغيل
المشروع محليًا.
